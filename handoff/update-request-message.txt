XXXBER 本番更新 引き継ぎ文

お疲れさまです。
XXXBERの売上管理アプリで、複数端末から入力した時に同期されない、入力内容が反映されない、伝票を作っても保存できないことがある、という問題が出ていたため、修正版を作成しました。

このフォルダ一式をGitHubの `nextonestepinfo-ops/XXXBER` にアップロードしてください。
最低限、本番修正だけならルート直下の `index.html` を置き換えれば反映されます。
`recovery/`、`handoff/`、`tools/` も一緒に入れると、リカバリー版、手順書、テストも残せます。

本番URL:
https://nextonestepinfo-ops.github.io/XXXBER/

主な問題

1. 保存APIの失敗を画面側が正しく判定していませんでした。
   以前はGASやスプレッドシート保存が失敗しても、通信を投げた時点で「保存しました」扱いになる可能性がありました。

2. 複数端末で未確定伝票を触った時、配列を丸ごと上書きするため、片方の端末の伝票がもう片方の保存で消えるリスクがありました。

3. 読み込みに失敗した端末や古い状態の端末が保存すると、サーバー側のデータを古い内容で上書きする可能性がありました。

4. 保存に失敗した時、現場側で未同期データを回収する導線が弱く、失敗データがどこにあるか分かりにくい状態でした。

修正内容

1. `saveData()` を厳格化しました。
   HTTPステータスとJSON本文を確認し、`success:true` 以外は保存成功扱いにしません。

2. 保存失敗時は「保存しました」を出さないようにしました。
   失敗したデータは端末内の `sales-manager-sync-outbox` に退避します。

3. 画面ヘッダーに `未同期 N件` を表示するようにしました。
   未同期がある場合は、再送ボタンとJSON出力で回収できます。

4. 未確定伝票の保存前に、サーバー最新版を取得してID単位でマージするようにしました。
   これにより、複数端末で作った未確定伝票が片方の保存で消えるリスクを減らしています。

5. 初期読み込みや再同期に失敗している端末では、保存を止めるようにしています。
   空データや古いデータで本番データを上書きしないためです。

6. CSV復元データや初期化処理は本番 `index.html` には入れていません。
   既存の本番スプレッドシートのデータを勝手に書き換えないため、コード修正だけに絞っています。

テスト済み内容

- 本番用自動テストを10回連続で実行して通過
- リカバリー用自動テストを5回連続で実行して通過
- 保存成功ケース
- 保存失敗ケース
- GASが `success:false` を返すケース
- 読み込み失敗時に保存を止めるケース
- 未同期データの再送
- 2端末相当の未確定伝票マージ
- 公開用ファイルにCSVの顧客名や売上明細を含めていないこと

アップロード前の注意

- できれば本番スプレッドシートを丸ごとコピーしてバックアップしてください。
- CSVバックアップは売上復元の保険になりますが、スタッフ、メニュー、未確定伝票、前借り、金庫、経費まで完全に戻せるとは限りません。
- 本番に反映する時は、CSVデータ投入や初期化はしないでください。
- 今回の本番更新は `index.html` のコード修正が中心です。

アップロード後に確認すること

1. 本番URLを開いて強制更新します。
2. ログイン後、ヘッダーに `同期エラー` が出ていないことを確認します。
3. `↻ 再同期` を押して、同期完了になることを確認します。
4. 2端末または2タブで未確定伝票をそれぞれ1件作ります。
5. 片方を保存しても、もう片方の未確定伝票が消えないことを確認します。
6. 会計確定を1件だけ試し、スプレッドシート側に反映されることを確認します。
7. 画面に `未同期 N件` が出た場合は、再送してください。
8. 再送しても残る場合は、`JSON出力` で未同期データを回収してください。

補足

今回の修正で事故率はかなり下がりますが、Google Sheets/GASに配列を丸ごと保存する構造は残っています。
完全な同時編集対策をする場合は、次の段階でSupabaseなどの行単位保存へ移行するのが安全です。
